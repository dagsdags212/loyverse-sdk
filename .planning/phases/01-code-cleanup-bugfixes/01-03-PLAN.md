<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SDK code → error propagation | Exception type changes can alter which errors propagate to callers and how they are handled |
| JSON parse → response handling | Changing catch from `Exception` to `json.JSONDecodeError` means non-JSON parse errors (e.g., MemoryError, KeyboardInterrupt) now propagate correctly instead of being silently swallowed |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Denial of Service | `exporter.py` exception tightening | mitigate | Changing catch types could theoretically cause unhandled exceptions to propagate. However, the new types (`ExportError`, `duckdb.Error`, `pl.exceptions.PolarsError`) are the only plausible exceptions from the wrapped operations. `ExportError` wraps to itself (D-01) — consistent with pipeline pattern. Post-change import smoke test (D-06) validates no syntax/breakage issues. |
| T-01-02 | Information Disclosure | `client.py` exception tightening | mitigate | Previously, any `Exception` from `resp.json()` was caught, which would silently swallow `MemoryError`, `KeyboardInterrupt`, etc. Changing to `json.JSONDecodeError` is MORE secure — only JSON parse failures are handled, critical errors propagate. Verified via `grep` for `json.JSONDecodeError` count = 2. |
| T-01-03 | Tampering | `exporter.py` `raise Exception` → `raise ExportError` | mitigate | `raise Exception(...)` at line 390 was inconsistent with the pipeline's own exception hierarchy. Changing to `ExportError` (D-05) makes the code consistent and gives callers a typed exception to catch. Verified via grep for `ExportError` count increase by 1. |
| T-01-SC | Tampering | Package installs | accept | No new packages installed in this phase. |
</threat_model>
